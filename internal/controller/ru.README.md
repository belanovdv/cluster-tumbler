# internal/controller

Цикл согласования и потребитель команд, работающие только на лидере. Оба компонента запускаются, пока данный узел удерживает lease лидерства в etcd.

### controller.go — цикл согласования

`Reconcile` итерирует каждую management group и:
1. Пропускает группы с `managed=false` — они находятся под ручным управлением администратора.
2. Проверяет отсутствие ожидаемых ключей состояния ролей (обнаружение потери агента).
3. Агрегирует actual/health отдельных ролей в actual/health уровня группы.
4. Записывает actual и health группы обратно в etcd только при изменении.

`applyPriorityPolicy` работает в обоих режимах — `manual` и `automatic`:
- **Active-active топология** (все группы имеют одинаковый приоритет): устанавливает `desired=active` всем управляемым (`managed=true`) группам.
- **Active-passive топология**: реализует двухфазное переключение — сначала ожидает, пока текущая активная группа перейдёт в `actual=passive|failed`, затем активирует целевую группу. Таймаут ожидания: `max(check_interval + converge + exec)` по ролям группы (запасной — 30 с).
- **Только в automatic режиме**: `handleAutoFailover` обнаруживает сбой группы с наивысшим приоритетом и меняет её приоритет местами со следующей доступной управляемой группой, инициируя перемаршрутизацию на следующем цикле.
- Группы с `managed=false` контроллером никогда не изменяются.

`writeDesiredIfChanged` всегда записывает `managed=true` — контроллер управляет только группами в своей зоне ответственности.

### commands.go — потребитель команд

`CommandConsumer` отслеживает префикс `commands/` в etcd и обрабатывает входящие команды:

| Команда | Действие |
|---|---|
| `promote` | Меняет приоритеты так, чтобы целевая группа стала наивысшего приоритета; контроллер применяет переключение на следующем цикле согласования |
| `disable` | Устанавливает `managed=false` для группы, сохраняя текущее значение `desired`; контроллер и воркеры прекращают управление |
| `enable` | Устанавливает `managed=true`, сохраняя текущее значение `desired`; возвращает группу под нормальное управление — обратная операция к `disable` |
| `reload` | Записывает `desired=passive, managed=true`; воркеры ролей повторно пытаются выполнить passive-конвергенцию |
| `force_passive` | Требует `managed=false` и `desired=active`; записывает `desired=passive, managed=true`, чтобы воркеры выполнили `set_passive` и остановили сервисы перед повторным включением в пул |

Каждая команда проходит через состояния `pending → running → completed|failed` и архивируется в `commands_history/{id}`.
